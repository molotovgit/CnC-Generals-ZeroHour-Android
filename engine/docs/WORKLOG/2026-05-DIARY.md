## 2026-05-11: Fix FPS uncapped and frame counter instability (Issue #132)

Fixed three critical timing bugs affecting both Generals and Zero Hour that caused FPS to be uncapped or frame counter to appear erratic on non-Windows platforms.

What was done:
- **FramePacer API guard**: identified `FramePacer.cpp` using Windows-only `timeBeginPeriod()` / `timeEndPeriod()` API without `#ifdef _WIN32` guards, causing undefined behavior on macOS/Linux. Added `#ifdef _WIN32` guards in constructor/destructor.
- **FPS limit validation**: identified `FrameRateLimit::wait()` missing validation for `maxFps == 0` or unreasonable values, allowing uncapped mode unintentionally. Added early-exit handling for invalid FPS targets.
- **FPS init logic fix** (Generals base root cause): found `GameEngine.cpp` using `if (m_useFpsLimit && m_framesPerSecondLimit <= 0)` which prevented setting default FPS when `m_useFpsLimit=FALSE`. Changed to `if (m_framesPerSecondLimit <= 0)` to always ensure a valid BaseFps (30) default, independent of flag state. This was why Generals base appeared uncapped while Zero Hour worked—Generals' INI loading left m_framesPerSecondLimit=0 with m_useFpsLimit=FALSE.
- ensured shared `Core/GameEngine/` fixes automatically apply to both Generals base and Zero Hour expansions.

Validation:
- compiled both GeneralsX and GeneralsXZH successfully without errors.
- manually tested GeneralsXZH launch and confirmed stable frame pacing.
- manually tested GeneralsX launch and confirmed stable frame pacing (previously uncapped, now fixed).
