## 2026-06-28: Fix Uncapped FPS Frame Limiter

- **Uninitialized Variable Fix**: Fixed an issue where the game would run at uncapped framerates (e.g. 130 FPS instead of 60 FPS) in both Generals base game and Zero Hour. Discovered that the `W3DView` implementation of `getTimeMultiplier()` returned `m_timeMultiplier`, which was never initialized in the constructor. Because `m_timeMultiplier` contained garbage memory, the condition `getTimeMultiplier() <= 1` evaluated to false, leading `TheFramePacer` to bypass the FPS limit by setting it to `UncappedFpsValue` (1,000,000). Initializing `m_timeMultiplier = 1;` in `W3DView` constructor ([W3DView.cpp](file:///Users/felipebraz/PhpstormProjects/pessoal/GeneralsX/Core/GameEngineDevice/Source/W3DDevice/GameClient/W3DView.cpp)) fixes the issue.
- **Uncapped FPS Fast-Path Fix**: Corrected the fast-path check in `FrameRateLimit::wait()` ([FrameRateLimit.cpp](file:///Users/felipebraz/PhpstormProjects/pessoal/GeneralsX/Core/GameEngine/Source/Common/FrameRateLimit.cpp)) from `maxFps > 1000000` to `maxFps >= 1000000`. This ensures `UncappedFpsValue` (which is exactly `1000000`) is correctly matched by the fast-path check, bypassing the unnecessary 1-microsecond spin-wait loop.

## 2026-06-15: Fix Video Audio Playback with MiniAudio

Refactored `MiniAudioStream` to implement a thread-safe streaming `ma_data_source`, resolving the video audio silence issues under the MiniAudio backend.

Root cause:
- The previous implementation attempted to accumulate all PCM data in a single `std::vector` and waited for the buffer size to stop growing before creating and starting the sound.
- However, during playback, FFmpeg decodes video and audio frame-by-frame. The buffer was constantly growing, which either prevented the sound from being created at all, or only played the first fraction of a second before falling completely silent.

Fix:
- Inherited `ma_data_source_base` in `MiniAudioStream` and registered custom `read`, `seek`, and `get_data_format` callbacks.
- Used `std::mutex` and `std::lock_guard` to protect the internal audio buffer from concurrent access between the decoder (main thread) and miniaudio (IO thread).
- Implemented `readPCM` to serve decoded frames to the miniaudio IO thread. When an underflow occurs, it fills the requested output buffer with silence (zeros) so playback continues continuously and smoothly.
- Implemented dynamic memory reclamation in `readPCM` to periodically prune the consumed buffer data once the read cursor exceeds 1MB.
- Updated `createSound` to initialize the sound directly from `base` via `ma_sound_init_from_data_source`.
- Modified `update` to build the sound immediately as soon as the first frame initializes the format, allowing real-time playback.

Validation:
- Verified compilation and linking under macOS (`macos-vulkan` preset).

## 2026-06-04: Constrain Language.ini fallback chain

Tightened the English stock fallback added in 2026-05-27 so it
only activates when needed, and stops breaking non-English

  effectively broken) and leaves official localizations alone.
- Before invoking the fallback, probe
  `Data\English\Language.ini` with `TheFileSystem->doesFileExist`
  and skip silently when the file is not present in the deploy.
- CI for any official language (brazilian, polish, german, etc.)
  now works regardless of whether the deploy ships English.

## 2026-06-03: Fix Cyrillic string formatting on macOS (Issue #144)

`vswprintf` in `UnicodeString::format_va`. Reporter confirmed

- macOS `vswprintf` returns -1 when the wide-char format string

  when the underlying font was correctly resolved to
  `Arial Unicode MS`.

- `Core/GameEngine/Source/Common/System/UnicodeString.cpp` -
  `format_va` now wraps `vswprintf` in
  `uselocale(newlocale(LC_CTYPE_MASK, "UTF-8", 0))` for the
  duration of the call, then restores the previous thread locale.
  Single static `locale_t`, no per-call allocation.

Diagnostic instrumentation kept in tree (per reporter request,
will revisit after more user testing):
